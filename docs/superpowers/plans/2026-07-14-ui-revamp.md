# SentinelNet UI Revamp Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking. Load `frontend-design:frontend-design` before any task that authors markup/CSS.

**Goal:** Restyle the entire SentinelNet dashboard to the new design language (grouped left-sidebar IA, reusable module components, new operations home), wiring every control to a real backend endpoint so no button is dead — front-end only, no backend changes.

**Architecture:** The whole UI is a single inline file `templates/dashboard.html` (~11.5k lines, inline `<style>` + inline `<script>`) served verbatim as a `FileResponse` from `GET /` — there is no static mount and there is no Jinja templating. All work stays inside that one file so no backend, no `StaticFiles` mount, and no CSP CDN allowlist change is required. We keep the existing `switchTab()` engine, the `data-i18n` system, and the RBAC gate classes (`requires-write`, `requires-admin`); we replace the flat `.tab-nav` strip with a grouped sidebar, add a reusable component-CSS layer, add a Home tab, and restyle each tab body in place while preserving the element IDs the existing JS reads/writes.

**Tech Stack:** Vanilla HTML/CSS/JS (no build step, no framework), Font Awesome 6.5.2 + Google Fonts (already CSP-allowlisted), FastAPI backend (untouched), Python `unittest` + `fastapi.testclient.TestClient` for verification.

## Global Constraints

- **Front-end only.** No route added/changed, no new endpoint, no `StaticFiles` mount, no CSP edit, no Python behavior change. If a task appears to need a backend change, stop and flag it.
- **One file.** All markup/CSS/JS edits land in `templates/dashboard.html`. No new served asset files.
- **No dead controls.** Every button/link/action must call an existing endpoint from the audit map (`docs/ui-revamp/button-endpoint-map.md`) or an existing JS function that does. A control with no real target is removed, not stubbed.
- **Preserve element IDs and function names** that existing JS depends on. Restyle wraps/reclasses DOM; it does not rename the IDs that `loadXxx()`/render functions query.
- **Preserve RBAC gating.** Keep `requires-write` / `requires-admin` classes on every gated control and nav entry. Never expose an admin/operator action to a viewer.
- **Preserve i18n.** Every user-visible string keeps or gains a `data-i18n` (or `data-i18n-placeholder` / `data-i18n-title`) key present in BOTH `it` and `en` maps in the `i18n` object (starts line 2344). New strings default English copy matching the prototypes; Italian translation added in the same task.
- **Design tokens are fixed.** Use the existing CSS custom properties (`--surface`, `--surface-2`, `--surface-3`, `--border`, `--primary`, `--lime`, `--success`, `--warning`, `--danger`, `--cta`, `--cta-text`, radii, `--font-*`). Do not introduce new hex colors. Reference: `sentinelnet-design-language.md`.
- **CSRF header.** Any state-changing `fetch` goes through the existing `apiFetch()` wrapper (it attaches the `X-Requested-With: SentinelNet` header and cookie auth). Do not hand-roll `fetch` for POST/PUT/DELETE.
- **Preview badge.** Client Map, Live Flows, Sites carry an amber `PREVIEW` badge in the sidebar (per design language + prototypes).
- **Rebuild after done.** Final task runs `pyinstaller SentinelNet.spec` (project rule) and `graphify update .`.

## Verification model (applies to every task)

There is no JS test runner. UI is verified two ways, both required per task:

1. **Automated (regression gate):** a Python `unittest` using `TestClient(app_server.app)` that fetches `GET /` and asserts the rendered HTML contains the expected structural markers (IDs, classes, `data-i18n` keys, gate classes) and does NOT contain removed markers. New assertions go in a new file `test_ui_revamp.py`. This runs headless and cheap. Pattern mirrors existing `test_observability_ui.py`.
2. **Manual (behavioral gate):** launch the app and click the tab, confirming it renders and every button triggers a real network call. Use the `/run` skill (or `python -c "import app_server; app_server.main()"` equivalent with `SENTINELNET_NO_BROWSER=true`, then load `http://127.0.0.1:<port>/`). Confirm in the browser Network panel that each button hits a `/api/...` path from the audit map.

Run the full Python suite after each task:
```bash
uv run python -m unittest test_ui_revamp test_router_parity test_observability_ui -v
```
Expected: OK (no failures, no errors).

## Delegation model

This plan is executed via **subagent-driven-development**: one fresh subagent per task, two-stage review between tasks. Per-task delegation guidance:

- **Mechanical restyle tasks** (per-tab module conversion, wiring to known endpoints): delegate to a **sonnet** general-purpose subagent. Give it the task's endpoint-contract table + the ID-preservation list; it has zero prior context so the task must be self-contained (it is).
- **Design-judgement tasks** (component CSS layer, sidebar IA, Home tab): delegate to an **opus** subagent, or keep inline in the driving session.
- **Audit / read-only mapping** (Task 0): **sonnet** Explore-style subagent — pure grep/read, no edits.
- Each task below carries a **Delegation:** line naming the recommended subagent + model.
- Per project memory: prefer small agents for small tasks; keep patch files; no CI; rebuild exe at the end.
